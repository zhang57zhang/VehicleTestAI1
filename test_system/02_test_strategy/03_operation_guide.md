# 测试策略（TS）操作指导

## 文档信息

| 属性 | 内容 |
|------|------|
| 版本号 | V1.0.0 |

## 1. 概述

本文档为测试策略活动中的每个叶子节点提供标准操作指导，包括操作步骤、决策流程、常见问题处理和检查清单。操作指导旨在确保测试策略工程师在执行测试策略活动时能够遵循统一的标准流程，保证活动输出的质量和一致性。

## 2. 叶子节点标准操作步骤

### 2.1 测试范围定义子活动

#### ACT-TS-1-1-001 识别被测系统范围

**标准操作步骤**

| 步骤 | 操作内容 | 执行要点 | 交付物 |
|------|----------|----------|--------|
| 1 | 收集系统文档 | 收集系统架构设计文档、需求规格说明书等 | 系统文档收集清单 |
| 2 | 分析系统结构 | 理解系统的组成、模块划分和接口关系 | 系统结构分析报告 |
| 3 | 定义测试边界 | 明确测试涵盖的范围和不涵盖的范围 | 测试边界定义 |
| 4 | 识别外部交互 | 识别系统与外部环境的交互关系 | 外部交互清单 |
| 5 | 建立范围文档 | 编写被测系统范围定义文档 | 被测系统范围定义 |
| 6 | 评审确认 | 组织评审并获得确认 | 评审记录 |

**决策流程图**

```
开始 → 收集系统文档
          ↓
    分析系统结构
          ↓
    识别被测组件 → 组件是否在测试范围内？ → 否 → 排除 → 继续下一个
          ↓ 是
    定义测试边界
          ↓
    识别外部交互
          ↓
    建立范围文档
          ↓
    组织评审
          ↓
    评审通过？ → 否 → 修订 → 重新评审
          ↓ 是
    发布文档 → 结束
```

**常见问题处理**

| 问题 | 可能原因 | 处理方法 |
|------|----------|----------|
| 系统边界模糊 | 系统架构描述不清 | 协调系统工程师明确边界，必要时召开澄清会议 |
| 范围蔓延倾向 | 需求方期望过高 | 基于资源约束重新协商，明确优先级 |
| 接口信息不全 | 文档不完整 | 要求提供完整的接口规范，或通过接口分析补充 |

**检查清单**

- [ ] 系统文档已收集完整
- [ ] 系统结构已分析理解
- [ ] 测试边界已明确定义
- [ ] 外部交互已识别完整
- [ ] 范围定义文档已编写
- [ ] 评审已通过并确认

## 3. 质量保障措施

### 3.1 评审要求

所有关键交付物必须经过评审：

| 交付物类型 | 评审级别 | 评审人员 |
|------------|----------|----------|
| 测试策略文档 | 管理评审 | 项目经理、测试经理、质量经理 |
| 测试范围定义 | 技术评审 | 系统工程师、测试工程师 |
| 测试方法选择 | 技术评审 | 技术专家、测试工程师 |
| 测试资源规划 | 管理评审 | 项目经理、资源经理 |
| 测试风险分析 | 管理评审 | 项目经理、风险管理员 |
| 测试进度计划 | 管理评审 | 项目经理、干系人 |

### 3.2 文档质量标准

| 质量维度 | 标准要求 |
|----------|----------|
| 完整性 | 覆盖所有必要内容，无重大遗漏 |
| 正确性 | 内容准确，引用正确 |
| 一致性 | 内部一致，与其他文档一致 |
| 可追溯性 | 追踪关系完整清晰 |
| 可读性 | 表达清晰，结构合理 |
