# 需求分析（RA）操作指导

## 文档信息

| 属性 | 内容 |
|------|------|

## 1. 概述

本文档为需求分析活动中的每个叶子节点提供标准操作指导，包括操作步骤、决策流程、常见问题处理和检查清单。操作指导旨在确保测试工程师在执行需求分析活动时能够遵循统一的标准流程，保证活动输出的质量和一致性。

## 2. 叶子节点标准操作步骤

### 2.1 需求获取子活动

#### ACT-RA-1-1-001 识别法规标准需求

**标准操作步骤**

| 步骤 | 操作内容 | 执行要点 | 交付物 |
|------|----------|----------|--------|
| 1 | 收集适用法规清单 | 联系法务部门或标准查询网站，获取项目适用市场的法规清单 | 初步法规清单 |
| 2 | 筛选适用法规 | 根据产品类型和市场区域，筛选直接适用的法规 | 筛选后法规清单 |
| 3 | 分析法规条款 | 逐条分析适用法规，提取与控制器相关的条款 | 法规条款摘录 |
| 4 | 建立映射关系 | 将法规条款映射到系统需求 | 法规需求映射表 |
| 5 | 评审发布 | 组织内部评审，修订后发布 | 法规需求识别报告 |

**决策流程图**

```
开始 → 是否获取到法规清单？ → 否 → 联系法务/标准部门 → 返回步骤1
                          ↓ 是
                    分析法规条款
                          ↓
              是否有与控制器相关条款？ → 否 → 标记为不适用 → 继续下一条
                          ↓ 是
                    提取相关条款
                          ↓
              是否有解读疑问？ → 是 → 咨询法规专家 → 返回分析
                          ↓ 否
                    建立条款映射
                          ↓
                  完成所有条款？ → 否 → 继续分析 → 返回步骤3
                          ↓ 是
                    组织评审
                          ↓
                      评审通过？ → 否 → 修订问题 → 重新评审
                          ↓ 是
                    发布报告 → 结束
```

**常见问题处理**

| 问题 | 可能原因 | 处理方法 |
|------|----------|----------|
| 法规版本更新 | 法规标准定期修订 | 建立版本跟踪机制，每次分析前确认使用最新版本 |
| 多市场法规冲突 | 不同市场要求不一致 | 记录冲突点，提交产品决策委员会决定 |
| 法规解读歧义 | 法规条文表述模糊 | 咨询法规专家或权威解释，保留咨询记录 |
| 法规覆盖不完整 | 初步筛选遗漏 | 与行业协会联系，获取补充信息 |

**检查清单**

- [ ] 已获取项目目标市场的完整法规清单
- [ ] 已筛选直接适用的法规标准
- [ ] 已逐条分析法规条款并提取相关信息
- [ ] 已建立法规条款到需求的映射关系
- [ ] 已识别可能影响测试策略的特殊要求
- [ ] 法规需求识别报告已通过评审

#### ACT-RA-1-1-002 识别客户需求

**标准操作步骤**

| 步骤 | 操作内容 | 执行要点 | 交付物 |
|------|----------|----------|--------|
| 1 | 接收客户文档 | 接收并登记客户提供的所有需求文档 | 文档接收记录 |
| 2 | 阅读理解 | 仔细阅读所有客户文档，标注关键需求点 | 阅读笔记 |
| 3 | 需求分类 | 将客户需求按功能、性能、接口、质量分类 | 需求分类表 |
| 4 | 需求编号 | 为每个客户需求分配唯一标识符 | 编号后需求清单 |
| 5 | 冲突识别 | 识别需求间的冲突或矛盾 | 冲突清单 |
| 6 | 问题整理 | 整理需要向客户澄清的问题 | 问题清单 |
| 7 | 内部评审 | 组织内部评审需求分析结果 | 评审记录 |
| 8 | 客户确认 | 与客户确认需求理解正确性 | 客户确认签字 |

**决策流程图**

```
开始 → 接收客户文档
          ↓
    完整阅读文档
          ↓
    识别关键需求点
          ↓
    需求分类
          ↓
    是否有歧义需求？ → 是 → 记录问题
          ↓ 否
    是否有需求冲突？ → 是 → 记录冲突
          ↓ 否
    需求清单整理
          ↓
    内部评审
          ↓
    评审通过？ → 否 → 修订
          ↓ 是
    客户确认
          ↓
    客户确认通过？ → 否 → 澄清问题
          ↓ 是
    发布需求清单 → 结束
```

**常见问题处理**

| 问题 | 可能原因 | 处理方法 |
|------|----------|----------|
| 客户需求模糊 | 客户描述不精确 | 准备问题清单，与客户进行澄清会议 |
| 需求冲突 | 不同文档或客户不同部门的要求不一致 | 记录冲突，请求客户明确优先级 |
| 需求遗漏 | 客户提供文档不完整 | 与客户沟通补充，参考类似项目 |
| 需求变更频繁 | 客户需求不稳定 | 建立需求变更管理流程，记录变更历史 |

**检查清单**

- [ ] 已接收并登记所有客户需求文档
- [ ] 已完整阅读并理解所有客户需求
- [ ] 需求已按功能/性能/接口/质量分类
- [ ] 每个需求已分配唯一标识符
- [ ] 需求歧义已识别并记录
- [ ] 需求冲突已识别并记录
- [ ] 待澄清问题已整理
- [ ] 内部评审已通过
- [ ] 客户已确认需求理解正确

#### ACT-RA-1-1-003 识别内部设计需求

**标准操作步骤**

| 步骤 | 操作内容 | 执行要点 | 交付物 |
|------|----------|----------|--------|
| 1 | 收集架构文档 | 从配置库获取系统架构设计文档 | 架构文档收集记录 |
| 2 | 理解系统结构 | 分析系统架构，理解子系统划分 | 架构理解备忘录 |
| 3 | 提取接口需求 | 从架构文档中提取内部接口需求 | 接口需求清单 |
| 4 | 提取约束条件 | 提取设计约束（硬件、软件、资源限制） | 约束条件清单 |
| 5 | 提取复用需求 | 识别可复用组件及其需求 | 复用需求清单 |
| 6 | 建立映射 | 建立内部需求与客户需求的映射 | 需求映射表 |
| 7 | 评审发布 | 组织评审并发布 | 评审通过后发布 |

**常见问题处理**

| 问题 | 可能原因 | 处理方法 |
|------|----------|----------|
| 架构文档不完整 | 架构设计尚未完成 | 协调架构团队优先完成关键部分，或使用预估设计 |
| 接口定义变化 | 架构迭代导致接口调整 | 保持与架构团队沟通，及时更新内部需求 |
| 约束条件冲突 | 不同约束条件相互矛盾 | 提交架构团队决策，确定优先级 |

**检查清单**

- [ ] 已收集完整的系统架构文档
- [ ] 已理解系统结构和子系统划分
- [ ] 内部接口需求已提取完整
- [ ] 设计约束条件已整理
- [ ] 复用需求已识别
- [ ] 与客户需求的映射关系已建立
- [ ] 内部设计需求清单已通过评审

**HARA分析模板**

```markdown
危害分析与风险评估（HARA）

项目：[项目名称]
控制器：[控制器名称]
分析日期：[日期]
分析人员：[姓名]

| 序号 | 危害事件 | 运行场景 | 严重度S | 暴露率E | 可控性C | ASIL | 安全目标 |
|------|----------|----------|---------|---------|---------|------|----------|
|      |          |          | [0-3]   | [0-4]   | [0-3]   |      |          |

注：ASIL = S + E + C
- S=0: 无伤害
- S=1: 轻伤
- S=2: 严重伤害
- S=3: 致命伤害

- E=0: 从不
- E=1: 很低
- E=2: 低
- E=3: 中等
- E=4: 高

- C=0: 可控
- C=1: 简单可控
- C=2: 正常可控
- C=3: 不可控

ASIL判定：
- ASIL A: 最低安全要求
- ASIL B: 中等安全要求
- ASIL C: 较高安全要求
- ASIL D: 最高安全要求
```

## 3. 质量保障措施

### 3.1 同行评审要求

所有关键交付物必须经过同行评审：

| 交付物类型 | 评审级别 | 评审人员 |
|------------|----------|----------|
| 需求分析报告 | 技术评审 | 系统工程师、功能安全工程师 |
| 需求模型 | 技术评审 | 系统工程师、软件架构师 |
| 追踪矩阵 | 管理评审 | 测试经理、配置管理员 |
| 评审材料 | 同行评审 | 测试团队成员 |

### 3.2 文档质量标准

| 质量维度 | 标准要求 |
|----------|----------|
| 完整性 | 覆盖所有相关需求，无重大遗漏 |
| 正确性 | 内容准确，引用正确 |
| 一致性 | 内部一致，与其他文档一致 |
| 可追溯性 | 追踪关系完整清晰 |
| 可读性 | 表达清晰，结构合理 |

### 3.3 配置管理要求

| 活动 | 配置项 | 管理要求 |
|------|--------|----------|
| 文档创建 | 所有分析文档 | 纳入配置管理 |
| 文档修订 | 修订版本记录 | 版本控制 |
| 文档发布 | 发布版本 | 基线管理 |
| 变更管理 | 变更申请和审批 | 变更控制流程 |
