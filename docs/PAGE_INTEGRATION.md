# 子页面挂接方式与代码对接规范

## 一、页面挂接机制概述

本系统采用模块化的插件式架构设计，所有子页面通过统一的接口规范与主框架进行挂接。这种设计确保了页面的独立性和可扩展性，同时保证了系统整体的协调运行。

### 1.1 核心挂接原理

每个子页面本质上是一个独立的JavaScript模块，通过以下三个核心机制与主框架连接：首先，页面通过`getPageContent(pageId)`方法返回HTML字符串，主框架将其注入到页面容器中；其次，页面通过全局`app`对象访问系统功能，包括文件操作、AI调用、状态管理等；第三，页面通过事件机制与主框架进行双向通信，实现数据的实时同步和状态更新。这种设计使得每个页面都可以独立开发、测试和部署，同时又能无缝集成到主系统中。

### 1.2 页面生命周期

页面的完整生命周期包含五个关键阶段：初始化阶段（页面首次创建时触发，用于初始化状态和加载资源）、挂载阶段（DOM元素渲染完成后触发，用于绑定事件和初始化UI组件）、激活阶段（用户切换到该页面时触发，用于加载数据和恢复状态）、失活阶段（用户切换到其他页面时触发，用于保存状态和释放资源）、销毁阶段（页面被关闭时触发，用于清理事件和释放内存）。主框架在切换标签时会自动管理这些生命周期钩子，确保页面状态的一致性和资源的合理利用。

## 二、代码对接方式详解

### 2.1 主框架接口

主框架通过全局`app`对象暴露以下核心接口，供所有子页面调用：

| 接口名称 | 参数 | 返回值 | 功能说明 |
|---------|------|--------|----------|
| `newProject()` | 无 | void | 打开新建项目对话框 |
| `openProject()` | 无 | void | 打开项目选择对话框 |
| `saveProject()` | 无 | void | 保存当前项目到本地存储 |
| `showModal(id)` | 模态框ID | void | 显示指定的模态框组件 |
| `closeModal(id)` | 模态框ID | void | 关闭指定的模态框组件 |
| `handleFileUpload(event, type)` | 事件对象、文件类型 | void | 处理文件上传逻辑 |
| `switchTab(pageId)` | 页面标识符 | void | 切换到指定标签页 |
| `showToast(message, type)` | 消息内容、类型 | void | 显示提示消息 |
| `renderPage(pageId)` | 页面标识符 | void | 重新渲染指定页面 |
| `currentProject` | 无 | Project对象 | 获取当前项目信息 |
| `uploadedFiles` | 无 | Map对象 | 获取已上传文件映射 |

### 2.2 页面配置注册

每个页面需要在主框架的`PAGE_CONFIG`中注册配置信息，配置项包括页面ID（唯一标识符）、页面名称（标签栏显示名称）、图标类名（FontAwesome图标类）、文件类型（关联的文件扩展名）、页面描述（鼠标悬停提示文本）、必需输入（执行AI所需的数据源）、可选输入（增强功能的补充数据源）、输出文件（生成的默认文件名）。配置示例如下：

```javascript
const PAGE_CONFIG = {
    'test-strategy': {
        name: '测试策略',
        icon: 'fa-chess',
        fileType: '.test-strategy.json',
        description: '定义测试策略和测试方法',
        requiredInputs: ['requirements'],
        optionalInputs: ['systemDesign', 'personnel', 'equipment'],
        outputFile: '测试策略_v{n}.json'
    }
};
```

### 2.3 数据流转机制

页面的数据流转遵循以下标准流程：用户操作触发页面事件处理器，事件处理器更新页面内部状态并将数据保存到项目存储中，同时通过`showToast()`向用户反馈操作结果，主框架监听状态变化并更新项目树显示。这种设计确保了数据的实时性和一致性，同时也支持撤销和重做功能。

## 三、测试策略页面实现详解

### 3.1 页面结构

测试策略页面是最完整的实现示例，包含以下功能模块：快速导入区提供四个快速上传入口，分别对应需求文档、系统设计、测试人员和测试设备四个数据源，每个入口都支持点击上传和拖拽上传两种方式；AI生成区通过模态框收集完整信息，包括必填的需求文档和多个可选输入项；结果展示区呈现AI生成的测试策略内容，支持复制、下载和优化操作；历史版本区记录策略的版本变更历史，支持版本对比和回滚功能。

### 3.2 核心功能实现

**文件上传处理函数**：`handleFileUpload(event, type)`函数负责处理用户上传的文件。函数首先检查文件对象是否存在，然后根据文件类型将文件存储到`uploadedFiles`映射中。接着更新对应的文件列表UI元素，显示文件名和文件大小。最后调用`showToast()`向用户反馈上传结果。这种设计使得文件上传的状态对用户完全透明，同时也为后续的AI生成提供了数据基础。

**测试策略生成函数**：`generateTestStrategy()`是页面的核心业务函数。函数首先检查必填项（需求文档）是否已上传，如果未上传则提示用户。然后显示加载遮罩，向用户说明AI正在处理。函数收集所有输入数据，包括上传的文件信息、选中的Skills选项、风险描述和工程师要求。数据收集完成后，函数调用AI Agent进行分析和处理。处理完成后，函数将结果格式化并显示在结果展示区，同时将结果保存到当前项目中。

**结果优化函数**：`optimizeResult()`函数支持用户对生成的测试策略进行优化。函数首先检查用户是否在AI交互框中输入了优化要求，如果没有输入则提示用户。确认有优化要求后，函数重新调用AI Agent，传入原始策略内容和用户的优化要求。AI Agent分析优化要求后，生成改进后的策略内容并更新显示。

### 3.3 模态框交互

测试策略页面使用三个模态框组件：第一个是测试策略生成模态框，包含完整的表单输入界面，用户可以上传文件、选择Skills、输入风险描述和工程师要求；第二个是结果展示模态框，显示AI生成的测试策略内容，提供复制、下载和优化按钮；第三个是优化输入模态框（集成在结果模态框中），用户在此输入优化要求，函数据此生成改进版本。所有模态框都支持ESC键关闭和点击外部关闭，提升用户体验。

## 四、其他15个页面的通用模式

### 4.1 需求分析页面

需求分析页面的核心功能是导入和管理测试需求文档。页面结构包括：文件上传区支持多文件上传，显示已上传文件的列表和状态；需求结构展示区以树形结构显示需求的层级关系；功能点统计区显示识别到的功能点数量和覆盖情况。AI集成点在于：AI解析功能可以自动从上传的文档中提取功能点，生成结构化的需求清单，识别需求之间的依赖关系和追溯关系。

页面与测试策略页面的差异在于：需求分析页面不需要输出文件到项目树，而是将提取的信息存入项目的需求部分；页面不需要单独的生成按钮，而是通过AI解析自动触发数据处理；页面的核心交互是文件列表的管理和需求结构的展示。

### 4.2 测试用例页面

测试用例页面的核心功能是管理和维护测试用例库。页面结构包括：用例统计区显示总用例数、自动化率、覆盖率等关键指标；用例列表区以表格形式展示所有用例，支持筛选、搜索和批量操作；用例编辑区支持新建和编辑单个用例。AI集成点在于：用例生成Agent可以根据需求和策略自动生成测试用例，用例优化Agent可以分析现有用例的覆盖情况并提出改进建议。

### 4.3 测试脚本页面

测试脚本页面的核心功能是管理和执行自动化测试脚本。页面结构包括：脚本库区以卡片形式展示不同类型的脚本集合，如HIL测试脚本、MIL测试脚本等；脚本详情区显示选中脚本的具体内容和执行历史；执行控制区提供执行、暂停、停止等控制按钮。AI集成点在于：脚本生成Agent可以将测试用例转换为可执行的自动化脚本，脚本优化Agent可以分析执行效率并提出优化建议。

### 4.4 看板类页面

看板类页面包括测试台架看板、测试进度看板、自动化看板和AI辅助看板。这些页面的共同特点是：主要展示实时数据和统计信息，不需要用户输入大量数据；页面内容定期自动刷新；关注数据的可视化呈现。测试台架看板显示各测试台架的实时状态和参数，测试进度看板显示整体项目进度和各阶段的完成情况，自动化看板显示CI/CD流水线的执行状态，AI辅助看板显示AI功能的使用统计和效率分析。

### 4.5 预留页面

待使用1至待使用5这五个页面是预留的扩展位。这些页面当前显示“功能开发中”的空状态，用户点击“建议此功能”按钮可以提交功能建议。设计上采用统一模板，包含页面标题、空状态提示和建议按钮，便于后续功能开发时快速替换为实际功能。

## 五、AI Agent集成规范

### 5.1 Agent调用模式

每个测试活动对应的AI Agent遵循统一的调用模式。调用前需要进行输入验证，检查必需的数据源是否已准备好；调用中需要显示加载状态，向用户反馈处理进度；调用后需要格式化结果并更新UI，同时将结果保存到项目存储中。

### 5.2 Agent类型定义

| Agent类型 | 输入数据 | 处理逻辑 | 输出结果 |
|-----------|---------|----------|----------|
| 需求分析Agent | 原始需求文档 | 解析、结构化、依赖分析 | 结构化需求清单 |
| 测试策略Agent | 需求、设计、风险、人员、设备 | 分析、规划、策略生成 | 测试策略文档 |
| 测试用例Agent | 需求、策略 | 用例设计、覆盖分析 | 测试用例集 |
| 测试脚本Agent | 用例、环境配置 | 脚本转换、适配 | 可执行脚本 |
| 报告生成Agent | 测试数据、评估结果 | 数据汇总、分析 | 测试报告 |

### 5.3 统一提示词模板

每个Agent使用统一的提示词模板结构。系统提示词定义Agent的角色和能力，上下文提示词提供项目背景和约束条件，任务提示词描述具体的执行要求，输出格式提示词指定返回结果的格式要求。这种设计确保了Agent输出的一致性和可控性。

## 六、文件存储与版本管理

### 6.1 项目文件结构

每个项目在本地存储中创建独立的命名空间，文件结构遵循预定义的层级关系。根目录包含项目配置文件`.project.json`，记录项目元数据和版本信息。各功能模块的输出文件存储在对应的子目录中，如策略目录存储测试策略文件，用例目录存储测试用例文件等。

### 6.2 版本管理机制

所有生成的文件采用版本号管理。文件名格式为`{功能}_{版本号}.{扩展名}`，如`测试策略_v1.json`、`测试策略_v2.json`。系统记录每次生成的时间戳和生成参数，便于用户追溯和对比。新版本生成时自动递增版本号，保留历史版本供用户查看和回滚。

### 6.3 项目树同步

当用户在页面中执行操作并生成新文件时，主框架自动更新左侧项目树的显示。项目树的更新包括：在对应的功能目录下添加新文件节点，如果是新版本则在文件节点旁显示版本号提示，如果操作失败则显示错误状态。这种实时同步确保用户始终看到项目的最新状态。

## 七、页面间数据共享

### 7.1 共享数据通道

页面之间通过`app.currentProject`对象共享数据。每个页面可以将需要共享的数据存储到`currentProject`的对应属性中，其他页面可以读取这些数据进行联动。例如，需求分析页面提取的需求信息可以被测试策略页面读取使用，测试策略页面的输出可以被测试用例页面作为输入参考。

### 7.2 数据同步事件

页面间的数据同步通过事件机制实现。当一个页面更新共享数据时，可以触发`projectUpdate`事件，其他监听该事件的页面可以自动刷新显示最新数据。这种设计避免了页面间的紧耦合，同时保证了数据的实时一致性。

## 八、扩展开发指南

### 8.1 新增页面流程

添加新的子页面需要以下步骤：首先在`renderTabs()`方法中添加页面标签配置；然后在`getPageContent()`方法中添加页面内容的获取逻辑；接着实现页面的HTML模板和交互逻辑；最后如果需要AI功能，则在相应的Agent中添加处理逻辑。

### 8.2 页面模板参考

新增页面时可以参考以下模板结构：

```javascript
renderXxxPage() {
    return `
        <div class="page-header">
            <h1 class="page-title">
                <i class="fas fa-icon" style="color: var(--primary);"></i>
                页面标题
            </h1>
            <p class="page-description">页面描述</p>
        </div>

        <div class="dashboard-grid" style="margin-bottom: 24px;">
            <div class="dashboard-card">
                <div class="dashboard-stat">数值</div>
                <div class="dashboard-label">标签</div>
            </div>
        </div>

        <div class="card">
            <div class="card-header">
                <span class="card-title">卡片标题</span>
                <button class="btn btn-primary" onclick="app.handleAction()">操作</button>
            </div>
            <div class="card-body">
                卡片内容
            </div>
        </div>
    `;
}
```

### 8.3 样式规范

所有页面使用统一的样式系统，包括：布局类（`dashboard-grid`、`card`、`page-header`）、组件类（`btn`、`form-input`、`table`）、状态类（`tag-success`、`tag-warning`、`tag-danger`）、辅助类（`empty-state`、`loading-overlay`）。遵循这些规范可以确保新页面与现有页面在视觉上保持一致。

## 九、常见问题处理

### 9.1 页面切换状态丢失

如果页面切换后状态丢失，需要检查是否在`onDeactivate`生命周期钩子中正确保存了状态，同时检查是否在`onActivate`生命周期钩子中正确恢复了状态。确保使用`app.currentProject`作为持久化存储而非页面局部变量。

### 9.2 文件上传失败

文件上传失败通常是因为文件大小超过限制或文件类型不支持。系统当前支持的最大文件大小为50MB，支持的格式包括PDF、Word、Excel系列。如果需要支持其他格式，可以在`handleFileUpload`函数中添加相应的处理逻辑。

### 9.3 AI生成超时

AI生成操作设置了30秒的超时限制。如果生成时间过长，可以考虑分步骤生成，先生成概要再生成详细内容，或者减少输入数据量以加快处理速度。如果频繁遇到超时问题，请检查网络连接状态。
